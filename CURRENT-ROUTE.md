## 当前主线

### M1：先打穿「同一个人」的最小闭环

Vela 当前不是做助手，不是做平台，不是做重资产 avatar 项目。

当前只做 5 件事：
1. **桌面聊天壳**：Electron + Web UI，能稳定打开使用
2. **连续人格**：固定人格种子 + 对话中不明显漂人格
3. **连续记忆**：本地记忆文件 + 会话摘要写入/读回
4. **语音模式**：按钮切换语音交互模式；关闭时退回 text-in / text-out
5. **轻量在场感**：占位 2D avatar / 极简状态 UI，先证明 speaking / listening / thinking / emotion 状态有用

## 当前冻结口径

### 产品边界
- 一期 = **纯聊天 avatar 伴侣**
- 不开放文件整理、系统操作、写代码、工具调用权限
- 二期才讨论“伴侣 + 轻助手”

### 技术边界
- 推理路线：**hybrid / local-first**
- 凡是涉及真实 LLM / provider / thinking / 记忆回灌 / 权限边界的整改，必须走：施工位自测 → 第二视角真实验收 → 主控 closure；主控不能兼任真实验收执行位
- 本地不可让渡资产：记忆文件、主动机制、关系/人格状态、avatar 控制层
- 系统中枢：只做 **Vela Core**，不做 OpenClaw 级完整 gateway
- 配置系统：**单一主配置文件** 做真源，不拆一地碎配置
- 运行数据 / 缓存 / 资产目录必须支持独立配置；当前默认优先落到 **D 盘**，避免 C 盘被运行时数据吃满
- 若沉淀出对 OpenClaw 有通用价值的源码，优先考虑抽象回馈，而不是无脑私有分叉
- `M1-T1 ~ M1-T5` 必须在**一个连续施工上下文**内推进，避免架构/人格/状态漂移
- 长任务默认派 **CLI 施工位**，不走零碎聊天式施工

### 审美边界
- UI 不允许有强 AI 工具味 / 后台管理台味
- 审美方向参考：**二次元乙游**
- 即使先用占位资产，也要优先保证“干净、轻、像产品”，不是技术 demo 味

### Avatar 边界
- 一期先用 **2D 占位 avatar / 极简状态 UI**
- 不碰 Live2D rigging
- 不碰 VRM 3D 主线
- 不做大动作库和复杂演出

### 语音边界
- 一期语音交互 = **按钮切换模式**
- 开发/验证可先用低成本/占位语音方案
- 正式 TTS 候选优先考虑 **MiniMax 同步 TTS WebSocket**

## 当前成功标准

满足以下条件，M1 才算成立：
- [ ] Electron 壳可稳定启动
- [ ] 文本聊天闭环成立
- [ ] 会话结束后本地能写记忆摘要
- [ ] 下次进入时能读回最近摘要，用户感觉“她记得上次聊过什么”
- [ ] 语音模式按钮可切换，关闭时退回文本模式
- [ ] avatar / 状态 UI 能表达最小在场感
- [ ] 用户主观觉得：她不像重新开机，更像同一个人

## 当前不做（碰了就算跑偏）

- 桌面整理
- 系统权限
- 编码助手能力
- 重型主动推送系统
- 复杂关系数值系统
- Live2D / VRM 重资产路线
- 多角色平台
- 成人向主线
- 完整 gateway / 插件平台
