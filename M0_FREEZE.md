# Vela M0 冻结版

## 0. 一句话拍板

**Vela 一期不是做最强 AI，也不是做万能助手，而是做一个“像同一个人”的 avatar 伴侣。**

核心是四个词：
- **连续人格**
- **连续记忆**
- **自然语音**
- **轻量在场感**

---

## 1. 为什么现在这样定

### 1.1 为什么不把 Vela 做成助手
因为产品灵魂会立刻变脏。

一旦一期开文件权限、系统权限、代码权限，产品重心会从：
- 陪伴感
- 人格感
- 关系连续性

迅速滑向：
- 任务执行
- 自动化
- 工具代理

那 Vela 就不再是虚拟伴侣，而会变成一个定位模糊的半助手半角色壳子。

**所以：一期必须先守住“伴侣第一，助手第二”。**

### 1.2 为什么不做纯聊天框
因为市面上已经有太多“会说话”的聊天框。

Vela 要成立，必须让用户感知到：
- 她记得我
- 她像同一个人
- 她不是每轮重新开机
- 她会在合适的时候提起旧事
- 她有一点身体感和在场感

如果做不到这些，Vela 就没有独立立项的必要。

### 1.3 为什么不做复杂系统
因为我们现在做的是 **M0 冻结**，不是“先把航母图纸全画完”。

如果一开始就做：
- 完整关系数值系统
- 复杂主动触达系统
- 重型 avatar runtime
- 多角色平台
- 通用 gateway

那项目会在还没验证核心价值前，就死在复杂度里。

**所以：M0 的任务是砍，不是堆。**

---

## 2. 项目定义（冻结）

### 2.1 Vela 是什么
Vela 是一个 **本地优先（local-first）的 avatar 伴侣系统**。

它的产品目标不是完成任务，而是建立持续的陪伴体验。

### 2.2 Vela 不是什么
Vela 一期明确不是：
- 万能助手
- 文件整理代理
- 系统操作代理
- 编码助手
- 自动化平台
- OpenClaw 分身
- 聊天框套皮

### 2.3 一期主定位
**纯聊天伴侣。**

一期只证明：
- 她是同一个人
- 她记得你
- 她能和你自然说话
- 她有轻量 avatar 反馈

---

## 3. 用户价值（冻结）

### 3.1 用户真正买单的不是“能力强”
而是这几个瞬间：
- 她记得上次聊过什么
- 她会主动延续未完的话题
- 她有稳定的说话方式和态度
- 她不像机器人重置
- 她会通过一点表情/动作让你觉得她在场

### 3.2 Vela 的核心卖点（冻结）
1. **连续人格**
2. **连续记忆**
3. **自然语音**
4. **轻量 avatar 在场感**

> 注意顺序：avatar 只是壳，不是灵魂；灵魂是人格和记忆。

---

## 4. 一期范围（冻结）

## 4.1 一期必须有

### A. 对话能力
- 文本聊天
- 语音聊天
- 云端主脑推理

### B. 人格连续性
- 稳定语气
- 稳定态度
- 不像每次重新开机
- 有可解释的喜恶，不随机抽风

### C. 记忆连续性
- 本地保存记忆文件
- 记住长期偏好
- 记住关键共同经历
- 能延续上次聊到一半的话题

### D. 轻量 avatar 反馈
- 3~5 个基础表情状态
- 说话/聆听/思考 3 类状态切换
- 少量极轻动作（可选：点头/轻微前倾）

### E. 轻主动性
- 启动时可自然接续旧话题
- 在会话内可主动提起相关旧事
- 不做后台骚扰型推送

---

## 4.2 一期明确不做

- 文件权限
- 系统权限
- 桌面整理
- 编码能力
- 多工具 agent
- 多角色平台
- 重型 Live2D / VRM 复杂动作库
- 完整好感度游戏系统
- 高复杂主动推送体系
- 成人向主线

> 原因：这些会直接把项目从“伴侣验证”拖进“平台开发/数值策划/自动化系统”。

---

## 5. 二期范围（仅作为方向，不施工）

二期才考虑：
- 伴侣 + 轻助手
- 弱桌面整理
- 更细的主动机制
- 更丰富的 avatar 反馈
- 更完整的关系系统

> 这部分 **不进入一期任务板**，只作为未来方向。

---

## 6. 技术路线（冻结）

## 6.1 总路线
**Hybrid / local-first**

### 本地负责
- 人格状态
- 关系状态
- 记忆文件
- 会话状态
- avatar 控制层
- 轻主动逻辑

### 云端负责
- 主脑推理
- 一期 ASR
- 一期 TTS

### 为什么这么做
- 纯本地：质量不稳，容易为了部署感牺牲主脑质量
- 纯云端：记忆主权、主动节奏权、伴侣连续性会变空
- 混合路线：保住灵魂层本地，同时用云端换取一期对话质量

---

## 6.2 客户端技术栈（冻结）
**Electron + Web UI**

### 为什么是 Electron
1. 你现有工程和思维栈已经偏 JS / Node，切过去最快
2. 本地文件读写、音频、常驻进程、状态持久化都顺手
3. 后面如果要加本地 daemon / tray / 定时器，演进自然
4. 比 Unity 轻；比 Tauri 更熟、更快落地；比纯浏览器壳更容易管本地状态

### 为什么不是 Unity
- 对一期来说过重
- 会把问题从“产品验证”带到“引擎工程”

### 为什么不是 Tauri
- 不是不能做，而是现在优先级不是“再省一点壳层资源”，而是“最快打穿核心体验”

---

## 6.3 Avatar 资产路线（冻结）
**一期先走“2D 静态 avatar + 表情状态切换”**

### 为什么这样定
这是为了刻意控复杂度。

一期 avatar 的任务不是炫技，而是完成：
- 在场感
- 说话状态感
- 情绪反馈感

这三个目标，用静态 2D 立绘 + 表情切换就能先证明 70%。

### 为什么不先上 Live2D
- 资产成本高
- rigging 成本高
- 容易把主线带回复杂美术管线
- 当前最需要证明的不是“她动得多漂亮”，而是“她像不像同一个人”

### 为什么不先上 VRM 3D
- 技术可行性已经被 demo 验过了，但 v0.1 上 3D 仍然比 2D 更重
- 会额外引入骨骼、动作、性能、资源管理复杂度

### 结论
- **M0 冻结：一期 avatar 先用 2D 表情态**
- VRM / 更复杂 avatar 作为后续增强路线，不进入一期验收主线

---

## 6.4 系统中枢（冻结）
**不做 OpenClaw 那种完整 Gateway，只做 Vela Core。**

### Vela Core 是什么
一个本地 companion core，负责：
1. 记忆读写
2. 人格/关系状态管理
3. 对话上下文组装
4. avatar 状态映射
5. 本地持久化

### 为什么不做完整 gateway
- 现在需求还没多入口、多 surface、多 agent 到那个程度
- 现在做 gateway 属于提前盖机场，不是解决核心体验
- 一期只需要一个稳定本地中枢，不需要平台级基础设施

---

## 7. 人格系统（冻结）

### 7.1 一期人格目标
- 语气稳定
- 会主动提起旧事
- 有自己的喜恶
- 关系可升可降，但不一键清零
- 不像每次重新开机
- 更像人，不像万能助手

### 7.2 一期人格实现原则
**不做复杂数值策划系统。**

一期只保留最小人格/关系骨架：
- `persona_profile`：固定人格底色
- `preference_profile`：学习到的用户偏好
- `relationship_state`：轻量关系状态（先不要拆 4 维复杂模型）

### 为什么不做复杂关系数值系统
因为一旦拆成：
- 熟悉度
- 信任度
- 亲近度
- 安全感

你实际上就在做一个数值策划项目。M0 不能放任这个坑打开。

### 一期最小做法
关系只做三档：
- `reserved`
- `warm`
- `close`

再加一个：
- `hurt_but_connected`（受伤但未断联）

这样就能表达：
- 会升温
- 会降温
- 有最低值
- 不重新开机

而不会掉进复杂度深坑。

---

## 8. 记忆系统（冻结）

### 8.1 一期记忆目标
不是“记很多”，而是“记对、能延续”。

### 8.2 一期最小记忆结构
只保留 3 类：
1. **Profile Memory**
   - 用户长期偏好
   - 长期身份信息
2. **Relationship Memory**
   - 重要共同经历
   - 对关系有影响的事件
3. **Session Summary Memory**
   - 每次会话后的简要摘要

### 8.3 一期记忆文件建议
本地目录建议：
- `profile.json`
- `relationship.json`
- `sessions/YYYY-MM-DD.jsonl`
- `memory-summary.json`

### 8.4 一期记忆机制
- 每轮对话结束后做**摘要写入**
- 下轮会话开始时读回最近若干摘要
- 先不做复杂衰减
- 先不做复杂权重系统
- 先不做复杂主动召回池

### 为什么这么做
因为一期真正要证明的是：
**她记得上次聊过什么。**

不是证明：
**她拥有一个最先进的长期记忆数据库。**

---

## 9. 主动机制（冻结）

### 9.1 一期只做“轻主动”
允许的主动行为：
- 启动时自然续接旧话题
- 会话中自然提起相关旧事
- 对上轮未完话题做轻量回访

### 9.2 一期不做
- 系统通知推送
- 后台打扰
- 高频 heartbeat 巡逻
- 复杂 cron 调度体系

### 为什么
主动机制一旦做重，极容易从“陪伴”变“骚扰”。
一期先在会话内证明主动感，而不是在系统级证明通知能力。

---

## 10. 语音机制（冻结）

### 一期要求
- 语音输入自然可用
- 语音输出自然可用
- 能显著提升“陪伴感”

### 一期路线
- ASR：云端
- TTS：云端
- 说话状态驱动 avatar 表情切换

### 为什么
因为一期目标不是证明“离线语音栈很强”，而是证明“和她说话像在和一个存在交流”。

---

## 11. MVP 验收标准（冻结）

以下 6 条满足 4 条以上，且前 3 条必须满足，才算 v0.1 成立：

### 必须满足
1. 连续使用数天后，用户主观明确感觉：**她记得自己**
2. 用户主观明确感觉：**她不像每次重新开机**
3. 用户主观明确感觉：**她更像伴侣，而不是聊天框**

### 其余目标
4. 语音聊天达到“自然可继续使用”的程度
5. 2D avatar 表情状态能增强在场感
6. 她能自然接续上轮或近期话题，而不是生硬检索式提旧事

## 12. 一期任务里程碑（冻结）

### M0 - 定义冻结（当前）
输出：
- 项目定义
- 范围边界
- 技术路线
- 复杂度红线

### M1 - Core 闭环
目标：
- Electron 壳
- 文本聊天

### M2 - Voice 闭环
目标：
- ASR/TTS 接入
- 语音聊天闭环

### M3 - Avatar 闭环
目标：
- 2D avatar 表情态
- 状态联动
- 最轻在场感成立

### M4 - 轻主动闭环
目标：
- 启动续接旧话题

## 13. 复杂度红线（冻结）

一期任何需求只要踩中以下任一条，默认打回：

1. 把 Vela 往万能助手方向拉
2. 把 avatar 做成重资产演出项目
3. 把记忆系统做成复杂数据库研究项目
4. 把关系系统做成数值策划游戏
5. 把系统中枢做成完整 gateway / 平台基础设施
6. 把语音栈做成纯本地攻坚项目

### 建议直接拍板通过
1. 一期 = 纯聊天伴侣
2. hybrid / local-first
3. Electron + Web UI
4. 2D avatar 表情态优先
5. 不做完整 gateway，只做 Vela Core

如果你通过，这些就进入后续文档和任务板，不再来回漂。

## 15. 最终一句话

**Vela M0 的本质，不是在设计一个大系统，而是在保护一期只做最值钱的东西：让她像同一个人。**
